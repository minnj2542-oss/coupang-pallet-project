# 프로젝트: Coupang Supplier Hub 트럭적재(파레트) 자동구성 웹앱

## 목적
쿠팡 트럭 쉽먼트 시, 상차리스트(출고할 상품 목록)를 업로드하면 쿠팡 규격에 맞춰
파레트(PLT) 구성을 자동으로 계산해주는 단일 HTML 웹앱. (index.html 하나로 동작, 서버 없음)

## 핵심 파일
- `index.html` : 전체 앱 (HTML+CSS+JS 한 파일). localStorage로 마스터데이터 영구저장 (2026-07-12부터, GitHub Pages 배포 대비 window.storage에서 전환). SheetJS는 더 이상 사용 안 함(파일 업로드 기능 제거, 아래 참고).

## 도메인 규칙 (반드시 지킬 것) — v3 (2026-07-13 업데이트, 사용자와 대화로 확정)

1. **상차리스트(입력값)가 메인.** 상차리스트에는 이미 PLT 값이 포함되어 들어온다 (외부에서 계산되어 옴).
   열 순서 고정 8열 (2026-07-13부터, v3): **고객사&거래처명 / 상품코드 / 상품명 / BOX바코드 / 상품바코드 / 박스수 / 낱개수 / PLT**
   - `박스수`와 `낱개수`가 이제 분리된 별도 열이다 (이전엔 "수량" 하나로 낱개수만 있었음). 마스터 override 계산은 **박스수**를 쓰고(단위 정합성 확보, 예전 미해결 이슈 해소됨), 파레트 상세 화면 표시는 여전히 **낱개수** 기준.
   - `고객사&거래처명`(예: "동탄1", "이천2")은 배차번호 붙은 상차리스트에 여러 거래처가 섞여 들어올 때 쓰는 필터 키. 아래 규칙 7 참고.
   - GRID_COLS = `['vendor','code','name','b188','b88','boxQty','qty','plt']` 순서로 내부 필드명과 매핑됨.
   - **입력 방식은 엑셀 붙여넣기 전용 스프레드시트 그리드**. 사용자 회사 환경에서 파일 업로드가 막혀 있어 파일 업로드 기능은 제거함. `#ship-grid-tbody`에 30행짜리 빈 표를 기본 제공하고, 아무 칸에 포커스 후 Ctrl+V로 엑셀 셀 여러 개를 한번에 붙여넣으면 해당 칸부터 채워지고 모자란 행은 자동으로 늘어남 (`shipGrid` 상태, paste 이벤트 위임으로 처리). "상차리스트 불러오기" 버튼을 눌러야 `fullShipment` 배열로 확정됨 (거래처 필터링 전 전체).
2. **재적 마스터리스트는 보조 도구일 뿐.** 대부분(거의 전부)의 입력 상품은 마스터에 없고, 그 경우 입력된 PLT를 그대로 신뢰해서 사용한다. 마스터에 매칭되는 상품은 아주 소수(1~2개 수준)만 있을 것으로 예상.
   - 정확히 4열: 상품코드 / 상품명 / 박스수·PLT(자사 기준, 참고용) / 쿠팡박스수·PLT
   - "쿠팡박스수·PLT" 칸은 "48 (8방 * 6단)"처럼 숫자+구성설명이 한 칸에 같이 들어감. 계산 시엔 맨 앞 숫자만 파싱해서 사용 (cpBoxNum 함수).
   - 이 리스트는 현재 브라우저별 localStorage 저장 (팀 공유 아님, 2026-07-12 결정 — 배포 섹션 참고).
3. **PLT 결정 로직 (resolveItemPlt)**: 상품코드가 마스터에 등록되어 있고 쿠팡박스수/PLT가 파싱 가능하고 **박스수도 입력되어 있으면** → 입력된 PLT를 **무시**하고 `박스수 ÷ 쿠팡박스수/PLT`로 재계산해서 덮어씀 (override). 그 외에는 입력된 PLT 값을 그대로 사용 (pass-through).
   - 매칭 로직: 상품코드 우선 → 실패 시 상품명.
   - override가 적용된 상품만 화면에 별도로 표시 ("마스터 기준으로 재계산된 상품" 카드). 매칭 안 된 것은 이제 정상 케이스이므로 경고를 띄우지 않는다.
   - PLT 값이 비어있거나 숫자가 아니고, 마스터에도 없는 경우에만 계산에서 제외하고 "PLT 값 누락" 카드로 경고.
4. **파레트 구성 알고리즘**: 상품별로 결정된 PLT(occupancy)를 합산해서 사용. 마스터 매칭 여부와 무관하게 동일한 방식으로 처리한다.
   - 상품별 총 PLT(occupancy)에서 정수부만큼 완충(단일 SKU, 1.00 PLT) 파레트를 먼저 만든다.
   - 남는 소수부(remainder)는 여러 SKU를 묶어서 혼적 파레트로 구성한다.
   - 혼적 파레트(SKU 2개 이상)는 절대 **0.8 PLT**를 넘을 수 없다 (하드 캡, 2026-07-12 기준 0.85→0.8로 변경). 최대한 0.8에 가깝게 채우는 것이 이상적.
     - 이유: 다양한 상품을 섞어 짤 때 실제 1PLT 높이를 초과하지 않기 위한 안전마진.
   - 완충(단일 SKU) 파레트는 정의상 1.00(100%)이 정상이며 이 캡이 적용되지 않는다.
   - 혼적 파레트 SKU 최대 **7개** 제한 (2026-07-13, 10→7. 라벨 잘림 방지 + 뒷부분 SKU 과다 누적 방지 목적).
   - **Best-Fit-Decreasing** 방식 사용 (2026-07-13, First-Fit-Decreasing에서 교체). 자투리를 큰 것부터 정렬한 뒤, 들어갈 수 있는 기존 혼적 파레트 중 "넣었을 때 남는 공간(0.8-fill)이 가장 적은" 곳을 골라서 채움. 어디에도 못 들어가면 새 파레트 시작. First-Fit은 뒤쪽에 자투리가 몰려 0.8에 한참 못 미치는 파레트가 여러 개 남는 문제가 있었는데, Best-Fit은 대부분의 혼적 파레트를 0.75~0.8 사이로 촘촘히 채움 (실측: 40개 랜덤 데이터 기준 혼적 파레트 대부분 0.77~0.80, 평균 0.74).
   - 박스수(표시용, "낱개수" 컬럼) 표시는 "총 낱개수 ÷ 총 occupancy"로 역산한 비율을 사용.
5. 결과 화면: 파레트별로 PLT 소수값(예: 0.50)으로 표시 (퍼센트 아님). 상단에 "총 재적 PLT(합산)" 통계 표시.
6. **기본 화면**: 앱 진입 시 "상차리스트 입력 & 계산" 탭이 먼저 보인다. "재적 마스터관리"는 보조 탭.
7. **거래처 선택 (2026-07-13 추가)**: "상차리스트 불러오기" 클릭 시 그리드 전체를 `fullShipment`로 파싱한 뒤, `vendor`(고객사&거래처명) 값의 종류가 2개 이상이면 계산으로 바로 안 가고 `#calc-vendor-view`(거래처 선택 화면)를 먼저 보여준다. 목록에서 거래처를 하나 클릭하면 `shipment = fullShipment.filter(그 거래처)`로 필터링해서 입력 화면(미리보기+계산 버튼)으로 돌아간다. 거래처가 1종류뿐이거나 비어있으면 선택 화면 없이 바로 진행. 여러 거래처를 한 번에 계산하는 기능은 없음(하나씩 선택 방식으로 사용자가 확정).
8. **계산 탭 내부는 입력→(거래처 선택)→결과→상세 화면 전환** (2026-07-12부터, 한 페이지에 다 쌓이면 헷갈린다는 피드백 반영):
   - `#calc-input-view` (입력) → "적재 계산하기" 클릭 → `#calc-results-view` (결과: override/missingplt 카드 + 통계 + 파레트 한 줄짜리 리스트) → 파레트 행 클릭 → `#calc-detail-view` (해당 파레트 상세, 전체 너비 표라 바코드도 안 잘림).
   - 항상 한 번에 하나의 뷰만 보이며(showInputView/showResultsView/openPalletDetail/closePalletDetail로 전환), 각 화면에 뒤로가기 버튼이 있음. 새 브라우저 탭/페이지가 아니라 같은 탭 안에서 영역만 바뀌는 것.
   - 파레트 상세 화면엔 "← 이전 파레트/다음 파레트 →" 버튼과 "N / 총개수" 표시가 있어 목록으로 안 돌아가고 바로 옆 파레트로 넘어갈 수 있음 (경계에서는 버튼 비활성화, `currentDetailIdx` 전역변수로 위치 추적).
   - 스크롤 위치 규칙 (2026-07-12): "상차리스트 불러오기" 클릭 시 미리보기 카드로 스크롤(scrollIntoView) / "적재 계산하기" 클릭 시 결과 화면 맨 위로 스크롤(showResultsView 내부) / 파레트 상세 진입 시에도 맨 위로 / "← 목록으로" 클릭 시엔 맨 위가 아니라 방금 보던 파레트 행(`.pallet-row[data-idx]`)으로 스크롤. 전부 `behavior:'auto'` 사용(smooth는 이 환경 자동화 테스트에서 타이밍이 불안정해서 피함).
