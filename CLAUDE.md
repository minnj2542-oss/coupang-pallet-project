# 프로젝트: Coupang Supplier Hub 트럭적재(파레트) 자동구성 웹앱

## 목적
쿠팡 트럭 쉽먼트 시, 상차리스트(출고할 상품 목록)를 업로드하면 쿠팡 규격에 맞춰
파레트(PLT) 구성을 자동으로 계산해주는 단일 HTML 웹앱. (index.html 하나로 동작, 서버 없음)

## 핵심 파일
- `index.html` : 전체 앱 (HTML+CSS+JS 한 파일). localStorage로 데이터 영구저장 (2026-07-12부터, GitHub Pages 배포 대비 window.storage에서 전환). 저장 키 2개: `master-list`(재적 마스터, 계산 미사용) / `unit-reference-list`(단위 기준표, 계산에 사용). SheetJS는 더 이상 사용 안 함(파일 업로드 기능 제거, 아래 참고).

## 도메인 규칙 (반드시 지킬 것) — v3 (2026-07-13 업데이트, 사용자와 대화로 확정)

1. **상차리스트(입력값)가 메인.** 상차리스트에는 이미 PLT 값이 포함되어 들어온다 (외부에서 계산되어 옴).
   열 순서 고정 10열 (2026-07-13, v3.2로 확정): **고객사&거래처명 / 상품코드 / 상품명 / BOX바코드 / 상품바코드 / BOX입수 / BOX / EA / 총낱개 / PLT수**
   - `BOX입수`(박스당 낱개수) × `BOX`(박스수) = `총낱개`. `EA`는 실사용상 항상 0에 가까운 값이라(사용자 확인) 계산에 안 씀, 그리드에만 존재.
   - 마스터 override 계산은 **BOX**(박스수)를 쓰고(단위 정합성 확보), 파레트/미리보기 등 화면 표시 수량은 **총낱개** 기준. 총낱개는 낱개가 최소 단위라 **절대 소수점이 나올 수 없음** — 파싱 시(`importShipGrid`)와 파레트 배분 시(아래 규칙 4) 둘 다 정수로 강제함.
   - `고객사&거래처명`(예: "동탄1", "이천2")은 배차번호 붙은 상차리스트에 여러 거래처가 섞여 들어올 때 쓰는 필터 키. 아래 규칙 7 참고.
   - GRID_COLS = `['vendor','code','name','b188','b88','unitsPerBox','boxQty','eaExtra','totalUnits','plt']` 순서로 내부 필드명과 매핑됨.
   - **입력 방식은 엑셀 붙여넣기 전용 스프레드시트 그리드**. 사용자 회사 환경에서 파일 업로드가 막혀 있어 파일 업로드 기능은 제거함. `#ship-grid-tbody`에 30행짜리 빈 표를 기본 제공하고, 아무 칸에 포커스 후 Ctrl+V로 엑셀 셀 여러 개를 한번에 붙여넣으면 해당 칸부터 채워지고 모자란 행은 자동으로 늘어남 (`shipGrid` 상태, paste 이벤트 위임으로 처리). "상차리스트 불러오기" 버튼을 눌러야 `fullShipment` 배열로 확정됨 (거래처 필터링 전 전체).
   - **완전히 빈 행은 자동 무시**됨 (`importShipGrid`가 `r.code`가 없는 행을 필터링) — 이미 기본 동작이라 별도 처리 불필요함을 2026-07-14 확인.
2. **재적 마스터리스트** (`#panel-master`, `master` 배열, `MASTER_KEY` localStorage). **2026-07-14 사용자가 명확히 구분**: "재적"은 원래 목적(입고 파레트 → 쿠팡 출고 형태로 변환)을 위한 것이고, 아래 3번 "단위 기준표"와는 **완전히 다른 용도**. 지금은 계산 로직에서 전혀 안 쓰이는 순수 데이터 저장 탭 (향후 "재적" 관련 기능이 별도로 요청되면 그때 사용).
   - 정확히 4열 (원래 스키마로 복원, v3.3에서 한 번 잘못 바꿨다가 되돌림): 상품코드 / 상품명 / 박스수·PLT(자사 기준, 참고용) / 쿠팡박스수·PLT
   - 계산(resolveItemPlt)에 관여하지 않음. 절대 이 탭 스키마를 임의로 다시 바꾸지 말 것 — 바뀔 일 있으면 반드시 먼저 확인.
3. **단위 기준표** (`#panel-unitref`, `unitRef` 배열, `UNITREF_KEY` localStorage, 2026-07-14 신설). 회사가 취급하는 **전 상품**(~1200개)의 "PLT당 박스량"과 "박스당 낱개수"를 담는 참조 테이블.
   - 정확히 4열: 상품코드 / 상품명 / PLT당 박스량 / 박스당 낱개수
   - "PLT당 박스량" 칸은 "48 (8방 * 6단)"처럼 숫자+구성설명이 섞여 들어갈 수 있음. 계산 시엔 맨 앞 숫자만 파싱 (cpBoxNum 함수). "박스당 낱개수"는 순수 숫자.
   - **중요: 이 표는 PLT 값을 바꾸는 데 절대 쓰지 않는다.** `cpBox × unitsPerBox`(PLT당박스량 × 박스당낱개수)로 "완충파레트 1개 = 정확히 몇 개"를 구해서, 도메인 규칙 5번의 낱개수 배분 계산에만 조용히 쓰인다. 2026-07-14, 사용자가 "PLT를 절대 바꾸면 안 된다"고 명확히 정정 — 아래 규칙 4번 참고.
   - 브라우저별 localStorage 저장, 팀 공유 안 됨 (배포 섹션 참고). ~1200행 붙여넣기 실측 300행/24ms로 성능 문제 없음 확인.
4. **PLT 결정 로직 (resolveItemPlt)** — 2026-07-14 두 차례 정정 끝에 최종 확정 (v3.7):
   - **PLT가 0보다 큰 정상값이면 절대 안 건드리고 그대로 사용한다.** (입력값 우선 원칙, 안 바뀜)
   - **PLT가 0/빈값/음수면** (예: 엑셀에서 반올림되며 소수점 정보가 날아간 경우) → **단위 기준표에 등록된 상품이면 `BOX ÷ PLT당박스량`으로 계산해서 재적을 살려낸다.** "재적이 통째로 빠지는 것보다 단위 기준표로 복구하는 게 최우선"이라고 사용자가 명확히 함(2026-07-14) — 이게 v3.6(순수 pass-through)에서 다시 바뀐 지점.
   - 그래도 안 되면(PLT도 0/빈값이고 단위 기준표에도 없음) 그때만 계산에서 제외하고 "PLT 값 누락" 카드로 경고.
   - 이 fallback이 적용됐다는 표시(뱃지/카드)는 안 띄움 — 조용히 정확한 값으로 채워짐.
   - 정리하면: 완전 무결한 PLT는 절대 안 건드림 / PLT가 명백히 깨진 경우(0 이하)만 단위 기준표로 복구 시도 / 그것도 안 되면 경고. "PLT는 절대 안 바뀐다"는 원칙은 **PLT가 유효할 때만** 적용되는 것으로 최종 정리됨.
5. **파레트 구성 알고리즘**: 상품별로 결정된 PLT(occupancy)를 합산해서 사용. 단위 기준표 매칭 여부와 무관하게 동일한 방식으로 처리한다.
   - 상품별 총 PLT(occupancy)에서 정수부만큼 완충(단일 SKU, 1.00 PLT) 파레트를 먼저 만든다.
   - 남는 소수부(remainder)는 여러 SKU를 묶어서 혼적 파레트로 구성한다.
   - 혼적 파레트(SKU 2개 이상)는 절대 **0.8 PLT**를 넘을 수 없다 (하드 캡, 2026-07-12 기준 0.85→0.8로 변경). 최대한 0.8에 가깝게 채우는 것이 이상적.
     - 이유: 다양한 상품을 섞어 짤 때 실제 1PLT 높이를 초과하지 않기 위한 안전마진.
   - 완충(단일 SKU) 파레트는 정의상 1.00(100%)이 정상이며 이 캡이 적용되지 않는다.
   - 혼적 파레트 SKU **7개가 기본 목표치**(2026-07-13, 10→7. 라벨 잘림 방지 + 뒷부분 SKU 과다 누적 방지 목적)지만 하드 캡은 아님 — 아주 작은 자투리(0.01~0.1)가 너무 많아 7개로는 도저히 안 나눠질 때는 초과를 허용함 (아래 3단계 참고). **0.8 fill 캡은 어떤 경우에도 절대 넘지 않는 진짜 하드 캡.**
   - **자투리를 "큰 것(≥0.1)"과 "작은 것(<0.1)"으로 나눠서 2단계로 채운다** (2026-07-13, v3.1 — Best-Fit만으로는 작은 조각들이 뒤쪽 파레트 1~2개에 몰려서 SKU는 다 찼는데 PLT는 텅 빈 파레트가 남는 문제가 있어서 추가 개선):
     1. 큰 자투리(0.1 이상)는 Best-Fit-Decreasing: 정렬 후 "들어갈 수 있는 기존 혼적 파레트 중 남는 공간이 가장 적은" 곳부터 채움.
     2. 작은 자투리(0.1 미만)는 먼저 1)에서 만든 파레트들의 남는 gap에 Best-Fit으로 최대한 흡수시킴.
     3. 그러고도 남는 작은 자투리는, 필요한 새 파레트 개수를 미리 계산(`총량÷0.8`과 `개수÷7` 중 큰 쪽)해서 그만큼의 새 파레트를 만들어놓고 **라운드로빈**으로 순서대로 나눠 담음 (한 파레트를 SKU 7개까지 다 채우고서야 다음으로 넘어가는 방식이 아님).
   - **4) 병합(consolidation) 단계** (2026-07-13, v3.2): 위 1~3단계 후에도 저충전(약한) 혼적 파레트가 여러 개 남으면 서로 합쳐서 개수를 줄인다. 1차는 "합친 SKU 수가 7개 이하가 되는 조합만" 합치고, 그래도 약한 파레트가 남으면 2차로 **SKU 7개 제한을 무시하고**(0.8 fill 캡은 계속 지키면서) 합친다. 즉 SKU 7개는 "가능하면 지키는" 소프트 기준이고, 정말 안 되면 초과한다.
   - **완충 파레트 1개당 낱개수 계산 (2026-07-14 정정)**: 상품코드가 단위 기준표에 있으면 `cpBox × unitsPerBox`(PLT당박스량 × 박스당낱개수)로 **정확하게** 계산. 없으면 `Math.round(상품별 총낱개 ÷ 총occupancy)`로 근사. 어느 쪽이든 PLT(occupancy) 자체는 안 건드림 — occupancy는 그대로 입력값 기준. 잔여 파레트 몫 = `상품별 총낱개 - (완충파레트 낱개수 × 완충파레트 개수)`로 역산해서 항상 정수+오차 없이 합계가 정확히 맞아떨어짐 (절대 소수점 안 나옴).
     - 실측 검증(단위 기준표 매칭 시): PLT당박스량 70 × 박스당낱개수 21 = 완충파레트 1470개 정확 (입력 PLT는 1.15든 뭐든 그대로 유지, floor/remainder만 그 값 기준으로 계산됨).
     - 단위 기준표에 없을 때 폴백 실측: 총낱개 1680 ÷ PLT 1.15 → `Math.round`로 완충파레트 1461개(근사치, 어쩔 수 없음).
6. 결과 화면: 파레트별로 PLT 소수값(예: 0.50)으로 표시 (퍼센트 아님). 상단에 "총 재적 PLT(합산)" 통계 표시.
7. **기본 화면**: 앱 진입 시 "상차리스트 입력 & 계산" 탭이 먼저 보인다. 나머지 두 탭("단위 기준표", "재적 마스터관리")은 보조 탭.
8. **거래처 선택 (2026-07-13 추가)**: "상차리스트 불러오기" 클릭 시 그리드 전체를 `fullShipment`로 파싱한 뒤, `vendor`(고객사&거래처명) 값의 종류가 2개 이상이면 계산으로 바로 안 가고 `#calc-vendor-view`(거래처 선택 화면)를 먼저 보여준다. 목록에서 거래처를 하나 클릭하면 `shipment = fullShipment.filter(그 거래처)`로 필터링해서 입력 화면(미리보기+계산 버튼)으로 돌아간다. 거래처가 1종류뿐이거나 비어있으면 선택 화면 없이 바로 진행. 여러 거래처를 한 번에 계산하는 기능은 없음(하나씩 선택 방식으로 사용자가 확정).
9. **계산 탭 내부는 입력→(거래처 선택)→결과→상세 화면 전환** (2026-07-12부터, 한 페이지에 다 쌓이면 헷갈린다는 피드백 반영):
   - `#calc-input-view` (입력) → "적재 계산하기" 클릭 → `#calc-results-view` (결과: override/missingplt 카드 + 통계 + 파레트 한 줄짜리 리스트) → 파레트 행 클릭 → `#calc-detail-view` (해당 파레트 상세, 전체 너비 표라 바코드도 안 잘림).
   - 항상 한 번에 하나의 뷰만 보이며(showInputView/showResultsView/openPalletDetail/closePalletDetail로 전환), 각 화면에 뒤로가기 버튼이 있음. 새 브라우저 탭/페이지가 아니라 같은 탭 안에서 영역만 바뀌는 것.
   - 파레트 상세 화면엔 "← 이전 파레트/다음 파레트 →" 버튼과 "N / 총개수" 표시가 있어 목록으로 안 돌아가고 바로 옆 파레트로 넘어갈 수 있음 (경계에서는 버튼 비활성화, `currentDetailIdx` 전역변수로 위치 추적).
   - 스크롤 위치 규칙 (2026-07-12): "상차리스트 불러오기" 클릭 시 미리보기 카드로 스크롤(scrollIntoView) / "적재 계산하기" 클릭 시 결과 화면 맨 위로 스크롤(showResultsView 내부) / 파레트 상세 진입 시에도 맨 위로 / "← 목록으로" 클릭 시엔 맨 위가 아니라 방금 보던 파레트 행(`.pallet-row[data-idx]`)으로 스크롤. 전부 `behavior:'auto'` 사용(smooth는 이 환경 자동화 테스트에서 타이밍이 불안정해서 피함).
